# AgentX WhatsApp Companion

Control AgentX on your desktop from WhatsApp on your phone. No Twilio, no cloud, no webhooks. Runs entirely on your machine using `whatsapp-web.js` (the same protocol WhatsApp Web uses).

## How It Works

```
[WhatsApp on phone]  <->  [WhatsApp servers]  <->  [WhatsApp Web in headless Chromium on your desktop]
                                                      |
                                                      v
                                                [Node bot]
                                                      |
                                                      v
                                          [agentx.ps1 CLI in this repo]
```

## Prerequisites

- Node.js 18.17+
- PowerShell 7+ (`pwsh`) on PATH
- Working AgentX install in this repo (`.agentx/agentx.ps1` exists)

## Setup (one-time)

```powershell
cd companions\whatsapp
npm install
Copy-Item config.example.json config.json
# edit config.json: put YOUR phone number (digits only, with country code) in allowedNumbers
npm start
```

On first run, scan the QR code printed in the terminal:
1. Open WhatsApp on your phone
2. Settings -> Linked Devices -> Link a Device
3. Scan the QR

The session is cached in `.wwebjs_auth/`. Future runs skip the QR.

## Usage

Send any of these messages from your allowed number to your own WhatsApp account. Commands sent from your phone's native WhatsApp client in self-chat are accepted; messages generated by the linked web session are ignored so the bot does not react to its own replies.

| Message | What runs |
|---------|-----------|
| `help` | Show command list |
| `ready` | Priority work queue |
| `state` | All agent states |
| `status` | Quality loop status |
| `deps 42` | Dependencies for issue 42 |
| `workflow engineer` | Workflow steps for engineer |
| `ship 42` | Autonomous ship of issue 42 |
| `loop start "Fix login bug"` | Start a quality loop |
| `loop iterate "Tests pass"` | Record an iteration |
| `loop complete "All gates pass"` | Mark loop complete |
| `run engineer "Add /health endpoint"` | Run a specific agent |
| `ask "what should I work on next?"` | Default agent |
| `raw config show` | Pass raw args to the CLI |

## Security

- **Allowlist required**: only numbers in `allowedNumbers` can issue commands. Unknown numbers are silently dropped (no echo to avoid leaking that the bot exists).
- **Local only**: all command execution happens on your desktop. WhatsApp never sees the AgentX output beyond what is replied as a message.
- **Don't commit `config.json`** -- it is gitignored.
- **Don't commit `.wwebjs_auth/`** -- it contains your WhatsApp session.

## Operational Notes

- The companion is a foreground process. Run it under `pm2`, a Windows scheduled task, or a `Start-Process -WindowStyle Hidden` wrapper to keep it alive.
- Long-running CLI commands stream output back only after completion (the agentx CLI is one-shot). For live progress, watch the desktop terminal.
- WhatsApp limits message length; replies are auto-chunked at 3500 chars and CLI output is truncated to 6000 chars by default (configurable).
- If WhatsApp logs you out, delete `.wwebjs_auth/` and re-scan.

## Troubleshooting

- **QR doesn't appear**: ensure your terminal supports unicode block characters; otherwise try a different terminal.
- **`pwsh` not found**: install PowerShell 7 or set `AGENTX_PWSH=powershell.exe` in env.
- **CLI not found**: set `repoPath` in `config.json` to the absolute path of the AgentX repo.
- **Replies go to wrong chat**: you must send the command from a number in the allowlist; replies go back to that same chat.

## Voice Notes

Send a WhatsApp voice note to the bot. It will:

1. Download the PTT (push-to-talk) audio
2. Transcribe via OpenAI Whisper (`whisper-1` by default)
3. Route the transcript through the same command router as text

Behavior is controlled by config:

- `openaiApiKey` (or `OPENAI_API_KEY` env): required to enable transcription
- `whisperModel`: defaults to `whisper-1`
- `whisperLanguage`: optional ISO language hint (e.g. `en`)
- `voiceAutoExecute`: when `true` (default), the transcript runs immediately. When `false`, the bot replies with the transcript only and waits for you to send the command as text.

Examples:

- Say: *"ready"* -> bot runs `agentx ready`
- Say: *"ship issue forty two"* -> transcript is `ship issue forty two`. The router expects `ship 42`, so this works best when you say *"ship 42"* or *"raw ship 42"*. Use `voiceAutoExecute: false` if you want a confirmation step.
- Say: *"run engineer add a health endpoint"* -> bot runs `run engineer "add a health endpoint"` (the router treats everything after the agent name as the task).

Tip: Whisper handles natural numbers poorly for issue IDs. Speak digits clearly or use the confirmation mode.

## Push Notifications

The bot watches `.agentx/state/loop-state.json` and proactively messages you when the AgentX quality loop changes state. You will get a WhatsApp message when:

- A loop **starts** (`active=false -> true`)
- An **iteration** is recorded (with the iteration summary)
- The loop **completes** (`status -> complete`)
- The loop **status** changes between any other states

Configure under `notifications` in `config.json`:

```json
"notifications": {
    "enabled": true,
    "targets": ["14155551234"],
    "events": ["started", "iteration", "complete", "status"]
}
```

- `enabled`: master switch
- `targets`: numbers to notify; defaults to `allowedNumbers` if omitted
- `events`: subset of `started`, `iteration`, `complete`, `status`, `init`. Drop `iteration` if you only want start/end pings.

The watcher uses `fs.watch` on the state directory with a 750ms debounce; no polling unless `fs.watch` is unavailable on your filesystem (then it falls back to a 5s poll).
