## Context

AgentX agents currently cannot drive a real browser. This blocks several recurring tasks:

- **UX prototype validation**: HTML/CSS prototypes shipped by the UX Designer agent cannot be opened, screenshotted, or audited (axe-core) by the agent that produced them. WCAG 2.1 AA claims rely on manual checking.
- **Web research**: The Consulting Research and Architect agents can fetch URLs via HTTP but cannot execute JavaScript, follow client-side redirects, or interact with SPAs.
- **HTML output verification**: When agents produce dashboards, reports, or web-based artifacts, there is no way to confirm visual rendering or accessibility before handoff.

The WorkPilot review surfaced "browser automation as a first-class tool" as one of the highest-leverage adoptable concepts. The canonical local-first option is Playwright; the question is *transport*: how an AgentX agent invokes browser actions.

**Constraints:**
- Local-first: no hosted browser service in the default path.
- No new agent role -- this is a capability that existing agents (UX Designer, Consulting Research, Engineer for QA) call into.
- Skill-first: the deliverable is a SKILL.md that orients agents, plus the wiring choice this ADR makes.
- Works on Windows, macOS, Linux.
- Honors the planned security profile model (controlled / restricted modes must be able to lock down or disable browser automation).
- No bundled Playwright install in this slice -- adoption should be opt-in via a documented prerequisite.

## Decision

We will adopt **Option A: Microsoft's Playwright MCP server** as the transport for browser automation, with the SKILL.md providing the orientation, anti-patterns, and security guidance.

- The skill instructs agents to call browser tools through the Playwright MCP server (e.g., `browser_navigate`, `browser_click`, `browser_type`, `browser_screenshot`, `browser_snapshot_a11y`).
- AgentX does not bundle Playwright. The skill documents the install command (`npx @playwright/mcp@latest`) and tells the user how to add it to `.vscode/mcp.json`.
- The UX Designer agent gets the skill added to its "Skills to Load" table for prototype validation.
- Skills.md gets the skill listed in the directory and Quick Reference.
- A follow-up DevOps issue will wire Playwright MCP into the workspace `.vscode/mcp.json` template, gated on user opt-in.

## Options Considered

### Option A: Playwright MCP server (recommended)

**Description:** Use Microsoft's official `@playwright/mcp` server. AgentX agents call its tools through the existing MCP wiring already used by the VS Code extension and the planned daemon (ADR-341).

**Pros:**
- MCP is the existing tool layer. No new abstraction.
- Tools are namespaced (`browser_*`) and discoverable via the standard MCP capability list.
- Maintained by Microsoft -- aligned with broader Microsoft tooling AgentX already targets.
- Auto-reconnect, transport-agnostic (stdio/SSE), and Claude/Copilot/GenAI-host parity out of the box.
- Honors security profiles cleanly: the profile gate can disable the entire MCP server in restricted mode without per-tool changes.
- Same surface available through the daemon (ADR-341) without duplicate wiring.

**Cons:**
- Adds a Node-based MCP server as a documented prerequisite (one-time `npx` install).
- Tool surface evolves with upstream releases; we have to track its changelog.

**Effort**: S - **Risk**: Low

### Option B: Bundled Playwright Node script via `run_in_terminal`

**Description:** Ship a small Node script under `scripts/browser/` that wraps Playwright. Agents invoke it via `run_in_terminal` for each action.

**Pros:**
- No external MCP server requirement; everything in-repo.
- Full control over the tool surface.

**Cons:**
- Reinvents MCP. Every action is a fresh process spawn -- slow, no persistent browser context across calls.
- No structured tool schemas; agents have to parse stdout.
- We carry the maintenance burden of a Playwright wrapper that already exists upstream.
- Cross-platform PowerShell / bash quoting issues on every call.
- Doesn't compose with the daemon (ADR-341) cleanly -- daemon would still need to shell out per call.

**Effort**: M - **Risk**: Medium

### Option C: Hosted browser service (BrowserBase / Anchor / Steel)

**Description:** Use a hosted browser service. Agents call its REST/WebSocket API.

**Pros:**
- No local Playwright install.
- Captures screenshots into hosted storage.
- Scales to parallel sessions easily.

**Cons:**
- Violates AgentX local-first principle by default.
- Sends potentially sensitive content to a third party.
- Network dependency; offline use blocked.
- Per-session pricing introduces an unbounded cost surface for agents.
- Doesn't compose with the planned restricted security profile (deny-all-network).

**Effort**: S - **Risk**: High (policy + privacy)

## Rationale

Option A wins on every comparison axis except "zero prerequisite":

1. **Reuses MCP**: AgentX has already chosen MCP as its tool layer. Adding a new transport just for browsers fragments the tool model.
2. **Persistent browser context**: Real browser automation requires a long-lived browser process across calls (cookies, page state, tab handles). Option B cannot deliver this; Option A and C both can.
3. **Local-first**: Option C breaks the principle outright. Option A keeps the browser local.
4. **Security profile alignment**: One MCP server is one gate to flip in `restricted` mode. Option B would require per-call gating that duplicates the security profile logic.
5. **Maintenance**: We avoid owning a Playwright wrapper.

The cost is one documented prerequisite. We mitigate it by:
- Telling users explicitly in the SKILL.md.
- Filing a follow-up DevOps issue to wire it into `.vscode/mcp.json` opt-in.
- Letting the skill degrade gracefully: if the MCP server is not present, the agent reports the prerequisite instead of guessing.

## Consequences

### Positive
- UX Designer can self-validate prototypes (axe a11y audit, screenshot, keyboard navigation check) before handing off.
- Consulting Research and Architect can read JavaScript-rendered pages.
- Engineer / QA flows gain cheap end-to-end browser checks.
- Same wiring works inside VS Code today and inside the daemon (ADR-341) tomorrow.

### Negative
- One more MCP server in a fully wired workspace; users must install it manually until the follow-up DevOps issue lands.
- We must track upstream Playwright MCP changes and update the skill periodically.

### Neutral
- The skill becomes the orientation point for any future browser-related work.
- Restricted security profile must explicitly include the Playwright MCP server in its disable list.

## Implementation

1. Create the skill at `.github/skills/development/browser-automation/SKILL.md` with frontmatter, when-to-use, prerequisites, tool surface, workflows, anti-patterns, related skills.
2. Add `references/wcag-validation.md` for axe-core severity mapping to AgentX review findings.
3. Update `Skills.md` directory and Quick Reference rows.
4. Update `.github/agents/ux-designer.agent.md` "Skills to Load" table.
5. Track the follow-up DevOps issue to wire Playwright MCP into `.vscode/mcp.json` opt-in.

**Out of scope for this ADR:**
- Bundling Playwright with AgentX.
- A dedicated browser-agent role.
- Computer-use-style desktop automation.
