## Context

AgentX today runs only on user invocation: VS Code chat, CLI one-shots (`agentx run`, `agentx loop ...`), and GitHub Actions workflows. There is no persistent local process other tools can drive. As a result:

- Scripts, CI runners, second IDEs, and external automation cannot reach AgentX agents without spawning a fresh CLI process per call (slow, no streaming, no shared session state).
- The OpenAI Chat Completions ecosystem (curl, IDE plugins, eval harnesses, agent frameworks) cannot use AgentX as a drop-in model.
- Long-running orchestrations across multiple tools have no shared address to reach AgentX as a participant.

WorkPilot demonstrates the value of an always-on local agent surface. We want the same affordance, scoped to AgentX's local-first principles.

**Requirements:**
- Long-running local process that survives across editor sessions.
- HTTP surface with OpenAI-compatible `/v1/chat/completions` endpoint.
- Streaming responses (SSE) for chat-style consumers.
- Local-first by default: bind `127.0.0.1` only; explicit opt-in for `0.0.0.0`.
- Reuse existing AgentX primitives (loop state, sessions, handoffs, MCP wiring) instead of cloning them.

**Constraints:**
- AgentX is a VS Code extension + PowerShell CLI. We will not introduce a third runtime stack just for this surface.
- Cross-platform parity (Windows, macOS, Linux) is required.
- Must respect the security profile model planned in WorkPilot-inspired follow-up work (`open` -> `restricted`).
- No multi-tenant cloud hosting; no Microsoft Teams/Outlook bridge in this slice.

**Background:**
The VS Code extension already runs a Node host with full access to MCP servers, agent definitions, and skills. The PowerShell CLI already has a standalone agentic runner ([`.agentx/agentic-runner.ps1`](../../../.agentx/agentic-runner.ps1)) but no daemon mode. State directories under `.agentx/state/`, `.agentx/sessions/`, and `.agentx/handoffs/` are already file-backed and process-agnostic.

## Decision

We will ship the always-on surface as **Option B: a standalone Node host (`agentx serve`) packaged with the AgentX CLI**, sharing core libraries with the VS Code extension via a small TypeScript runtime workspace. The host:

- Runs as `agentx serve start|stop|status|restart` from the existing PowerShell CLI wrapper.
- Binds `127.0.0.1:<port>` by default, with bearer-token auth from `.agentx/config.json`.
- Exposes a small AgentX-native HTTP API (`/health`, `/v1/models`, `/agentx/sessions/*`, `/agentx/loop/*`) plus an OpenAI-compatible `/v1/chat/completions` endpoint with SSE streaming.
- Reuses the same agent loader, MCP wiring, and loop state code as the VS Code extension, extracted into a shared package under `vscode-extension/src/runtime/` (or a new `packages/runtime/`).
- Honors security profiles: `restricted` blocks remote bind and disables non-localhost MCP tools.

Manual switching to a specialist agent stays as a fallback; the always-on surface adds an additional access path, it does not replace VS Code or the CLI.

## Options Considered

### Option A: In-extension Node HTTP server

**Description:** Run the HTTP server inside the VS Code extension host. The extension activates on workspace open and starts listening.

**Pros:**
- Zero new runtime stack -- reuses the extension's MCP wiring directly.
- Easiest auth integration with VS Code's `SecretStorage`.
- Minimum packaging effort.

**Cons:**
- Lifecycle is bound to VS Code: closing the editor stops the server.
- Cannot serve users running only the CLI or headless CI.
- Extension host crashes take down the API surface.
- Conflicts with VS Code's reload-on-config and dev-host workflows.

**Effort**: S - **Risk**: Medium

### Option B: Standalone Node host (`agentx serve`)

**Description:** Ship a small Node entry point alongside the CLI. PowerShell wrapper (`agentx serve`) shells to it. Shares core libraries with the VS Code extension via a workspace package.

**Pros:**
- Lifecycle independent of VS Code; survives editor restarts.
- Same Node ecosystem the extension already uses -- MCP SDK, agent loader, JSON-Schema validators all reusable.
- Cross-platform: Node is already a soft prerequisite for the extension's builds.
- Easy to install via the existing AgentX install scripts.
- Works for headless CI runners and remote-tunnel scenarios.

**Cons:**
- Adds a Node runtime requirement to the CLI install path (currently PowerShell-only is enough for many flows).
- Requires careful refactor to extract shared libraries from the extension without breaking the extension build.
- Two processes (extension + serve) may both want to read/write `.agentx/state/` -- needs file-locking discipline already partially solved by loop state.

**Effort**: M - **Risk**: Medium

### Option C: Standalone PowerShell host

**Description:** Build the HTTP server in PowerShell, reusing `agentic-runner.ps1` plumbing. Use `System.Net.HttpListener` or `Pode`.

**Pros:**
- Zero new runtime requirement; PowerShell 7 is already required.
- Leverages existing PowerShell loop/state utilities directly.
- Smallest install footprint.

**Cons:**
- MCP SDK is Node/Python-first; PowerShell would need a separate transport implementation or shell-out per call (slow, fragile).
- SSE streaming in PowerShell is awkward and poorly tested.
- Cannot share code with the extension; doubles maintenance burden as the agent surface evolves.
- Concurrent request handling in PowerShell is more limited than Node's event loop.

**Effort**: L - **Risk**: High

## Rationale

We chose **Option B** because:

1. **Lifecycle independence**: The single hardest constraint is "survives across editor sessions," which Option A cannot meet.
2. **MCP reuse**: AgentX's value is its MCP-wired agent ecosystem. The Node MCP SDK is mature; PowerShell parity (Option C) would be a permanent tax on every new MCP tool.
3. **Code sharing with the extension**: Extracting agent loader + MCP wiring + loop state into a shared package serves both the extension and the daemon, reducing drift over time.
4. **Reasonable install cost**: Node is already required to build/test the extension. Making it a runtime requirement for the CLI is a documented one-time install, not a recurring cost.
5. **Defers cross-cutting work cleanly**: Security profiles, auth rotation, and remote bind opt-in are all addressed at one HTTP boundary instead of two.

**Key decision factors:** lifecycle independence, MCP SDK availability, code-sharing cost, cross-platform parity.

## Consequences

### Positive
- AgentX becomes addressable by any HTTP client; fits drop-in OpenAI-compatible tools.
- Shared runtime package reduces extension/CLI drift.
- Foundation for the WorkPilot-inspired security profile, audit log, and E-stop work.

### Negative
- New install step: Node runtime required for `agentx serve` (extension already needs it for builds, but headless-CLI users must install it).
- Two writers to `.agentx/state/` (extension + daemon) -- requires explicit file-locking convention for loop state.
- New attack surface: the HTTP listener must default to localhost-only and require bearer auth from day one.

### Neutral
- A new directory `packages/runtime/` (or equivalent) for shared libraries.
- New PID/log files under `.agentx/state/serve/`.

## STRIDE Analysis (HTTP Surface)

| Threat | Surface | Mitigation |
|--------|---------|------------|
| **S**poofing | Unauthenticated HTTP client impersonates user | Bearer token required on every request; token stored only in `.agentx/config.json` (chmod 600 / NTFS user-only ACL) |
| **T**ampering | MITM modifies streaming responses | `127.0.0.1` default avoids the network entirely; opt-in `0.0.0.0` requires TLS (cert path in config) |
| **R**epudiation | Caller denies invoking a destructive tool | Append-only audit log under `.agentx/audit/` (planned in companion issue) records caller token hash + tool + args |
| **I**nformation disclosure | Token logged in plaintext | Token never logged; auth middleware redacts `Authorization` header from all log sinks |
| **D**enial of service | Caller floods with concurrent loops | Per-token concurrency cap; queue with 429 + `Retry-After` semantics; max in-flight loops configurable |
| **E**levation of privilege | Tool mapping lets a chat completion run shell commands beyond the security profile | Security profile is enforced inside the existing tool-hook layer; HTTP layer only provides the request envelope |

Additional threats:

- **SSRF**: Tool calls that take URLs (browser-automation skill, future fetch tools) MUST honor the security profile's URL allowlist; the HTTP daemon does not relax this.
- **Command injection via OpenAI tool args**: Map OpenAI `tool_calls` to AgentX tools through the same JSON-Schema validators the extension uses. Reject calls whose schema fails validation; never `eval`/`exec` raw strings.

## Implementation

**Detailed technical specification**: [SPEC-341.md](../specs/SPEC-341.md)

**High-level implementation plan:**
1. Extract shared runtime (agent loader, MCP wiring, loop state, JSON-Schema validators) into `packages/runtime/`.
2. Implement `agentx serve` entry point with `/health` and `/v1/models` only (smoke).
3. Add `/agentx/loop/*` and `/agentx/sessions/*` (AgentX-native surface).
4. Add `/v1/chat/completions` with non-streaming and SSE-streaming responses.
5. Add bearer-auth middleware, security-profile gate, and audit-log emitter (companion issue dependency).
6. Documentation, CLI wrappers, and `agentx serve start|stop|status` lifecycle commands.

**Out of scope for this ADR (deferred to follow-up issues):**
- Audit log implementation details (depends on the WorkPilot-inspired audit-log issue).
- E-stop sentinel handling.
- Multi-tenant or remote hosting.
- Microsoft Teams / Outlook bridge.
